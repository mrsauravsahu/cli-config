# Plan: Shell Startup Time Improvements

## How startup works (execution order)

Every new shell sources in this order:

1. `profiles/mrsauravsahu/.zshrc` — sets env vars, PATH, sources dotfiles
2. `src/scripts/env.zsh` — fast, sets 3 exports (`CLI_CONFIG_PROGRAMS_CONF`, `CLI_CONFIG_TOOLS_LOCATION`, `CLI_CONFIG_CONF_LOCATION`)
3. `current/programs.conf.zsh` (autogenerated) — sources 6 conf files:
   - `antigen.conf.sh` → loads `antigen.zsh` → runs `antigen apply` with 6 plugins
   - `nvm.conf.sh` — sets `$NVM_DIR` and PATH only (nvm shell functions never loaded)
   - `pyenv.conf.sh` → `eval "$(pyenv init -)"` + `eval "$(pyenv virtualenv-init -)"`
   - `gvm.conf.sh` → lazyload (already deferred — good)
   - `tfenv.conf.sh` — just PATH additions
   - `ohmyposh.conf.sh` → `eval "$(oh-my-posh init zsh ...)"`
4. `autoload -Uz compinit && compinit` — always runs unconditionally
5. `eval $(thefuck --alias)` — spawns Python on every shell
6. `auto_tmux`, dotfiles sourcing, asdf hooks

## Bottlenecks

### 1. antigen — highest cost (~200–500ms+)

`antigen.conf.sh` loads `antigen.zsh`, resolves and applies 6 plugins on every shell:

```sh
antigen bundle qoomon/zsh-lazyload
antigen bundle zsh-users/zsh-autosuggestions
antigen bundle ptavares/zsh-direnv@main
antigen bundle autojump
antigen bundle zsh-users/zsh-syntax-highlighting
antigen bundle ohmyzsh/ohmyzsh plugins/git
antigen apply
```

Antigen resolves plugins at runtime on every shell. The migration plan to **zimfw** already exists in `ai/plans-zsh-modules-alternatives.md` — zimfw generates a static `init.zsh` at install/update time; shell startup just `source`s it with no runtime resolution.

**Fix**: complete the zimfw migration per `ai/plans-zsh-modules-alternatives.md`.

---

### 2. `eval "$(pyenv init -)"` + `eval "$(pyenv virtualenv-init -)"` (~100–200ms)

`current/conf/pyenv.conf.sh` forks pyenv twice to generate shell integration code. This runs on every shell even when Python is not needed.

**Fix**: replace with static PATH setup and lazy init:

```zsh
export PYENV_ROOT="$CLI_CONFIG_ROOT/current/pyenv"
export PATH="$PYENV_ROOT/bin:$PYENV_ROOT/shims:$PATH"

# Defer full pyenv init until first use
pyenv() {
  unfunction pyenv
  eval "$(command pyenv init -)"
  eval "$(command pyenv virtualenv-init -)"
  pyenv "$@"
}
```

The shims cover most day-to-day use; the function only forks pyenv when you explicitly call it.

---

### 3. `eval "$(oh-my-posh init zsh ...)"` (~30–80ms)

`current/conf/ohmyposh.conf.sh` forks the oh-my-posh binary on every shell to generate a zsh init script. The output is deterministic for a given theme — it never needs to regenerate unless the theme changes.

**Fix**: cache the generated init script at install time (or on theme change) and source the cache:

```zsh
# In ohmyposh.configure.zsh, write a loader instead of a direct eval:
OMP_CACHE="\${CLI_CONFIG_ROOT}/current/ohmyposh/.init.cache.zsh"
OMP_THEME_MARKER="\${CLI_CONFIG_ROOT}/current/ohmyposh/.current_theme"
if [[ ! -f "\$OMP_CACHE" || "\$(cat \$OMP_THEME_MARKER 2>/dev/null)" != "\$CLI_CONFIG_THEME" ]]; then
  \$CLI_CONFIG_ROOT/current/ohmyposh/oh-my-posh init zsh \\
    --config \$CLI_CONFIG_ROOT/current/ohmyposh/themes/\$CLI_CONFIG_THEME.omp.json \\
    > "\$OMP_CACHE"
  echo "\$CLI_CONFIG_THEME" > "\$OMP_THEME_MARKER"
fi
source "\$OMP_CACHE"
```

### 4. `eval $(thefuck --alias)` (~100–300ms)

`setup.programs-conf.zsh` writes `eval $(thefuck --alias)` into `programs.conf.zsh`. thefuck is a Python process — spawning it on every shell costs 100–300ms even when you never use it.

**Fix**: replace with a lazy-load wrapper in `setup.programs-conf.zsh`:

```zsh
# Replace:
echo eval "\$(thefuck --alias)" >> $CLI_CONFIG_PROGRAMS_CONF

# With:
cat >> $CLI_CONFIG_PROGRAMS_CONF << 'EOF'
fuck() {
  eval "$(thefuck --alias)"
  unfunction fuck
  fuck "$@"
}
EOF
```

---

### 5. `compinit` runs unconditionally (~20–50ms)

`programs.conf.zsh` always runs `autoload -Uz compinit && compinit`, regenerating the completion dump on every shell.

**Fix**: cache the dump and only rebuild when it is more than 24 hours old:

```zsh
# Replace in setup.programs-conf.zsh:
echo 'autoload -Uz compinit && compinit' >> $CLI_CONFIG_PROGRAMS_CONF

# With:
cat >> $CLI_CONFIG_PROGRAMS_CONF << 'EOF'
autoload -Uz compinit
if [[ -n ${ZDOTDIR:-$HOME}/.zcompdump(#qN.mh+24) ]]; then
  compinit
else
  compinit -C
fi
EOF
```

### 6. `find` + `xargs rm` runs on every shell (~5–20ms + file I/O)

`profiles/default/.zshrc` line 12 runs this on every shell start:

```zsh
find ${CLI_CONFIG_ROOT}/current -maxdepth 2 -type f -regex '.*zwc$' | xargs ${XARGS_OPTIONS} rm
```

This is a one-time cleanup that belongs in the install script, not the shell rc.

**Fix**: move it to `src/defs/install.zsh` or `src/defs/configure.zsh`, remove from `profiles/default/.zshrc`.

---

### 7. `CLI_CONFIG_ROOT` detected via forked processes every shell

Both zshrc profiles run:

```zsh
CLI_CONFIG_ROOT=$(ls -la ~/.zshrc | sed "s/^.*\->//" | awk -F '/' 'NF{NF-=3}1' 'OFS=/' | xargs)
```

This forks `ls`, `sed`, `awk`, and `xargs` on every shell. Since `~/.zshrc` is a stable symlink, the value is constant.

**Fix**: use zsh's native `$commands` and `realpath` / `${(%):-%x}` without forking:

```zsh
CLI_CONFIG_ROOT="${${${:-~/.zshrc}:A:h}:h:h}"
```

(`${...:A}` resolves symlinks natively in zsh; `:h` strips the last path component.)

---

### 8. nvm is broken as configured

`current/conf/nvm.conf.sh` adds `$NVM_DIR` to PATH but never sources `nvm.sh`, so the `nvm` command is unavailable. The lazyload line exists in `nvm.configure.zsh` but is commented out with a note about AI agents.

**Fix**: re-enable lazyload for nvm now that it's understood — the AI agent concern is about subshells not inheriting nvm; lazyload does not affect that:

```zsh
# In nvm.configure.zsh, replace the commented line with:
printf 'lazyload nvm node npm -- "source $NVM_DIR/nvm.sh"\n' >>$CONF
```
